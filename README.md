# LLM Wiki — 你的 AI 驱动研究知识库

> 基于 [Andrej Karpathy 的 LLM Wiki 模式](https://x.com/karpathy/status/1876166922370412860) 构建。
> LLM 负责维护，你在 Obsidian 中阅读和浏览。就像用 IDE 看代码一样看你的知识库。

## 核心理念

传统 RAG：每次提问都从原始资料中重新发现知识，没有积累。

**LLM Wiki 不同**：当你加入一份新资料时，LLM 会读取它、提取关键信息，然后**整合到已有的 wiki 中**——更新实体页面、修正主题摘要、标注矛盾、强化综合论述。知识只编译一次，持续保持最新。

## 目录结构

```
/
├── raw/                       ← 原始资料（不可变，LLM 只读不写）
│   ├── articles/              ←   网页文章 / 博客
│   ├── papers/                ←   学术论文 / 技术报告
│   ├── transcripts/           ←   视频 / 播客转录
│   ├── notes/                 ←   已有笔记导入
│   └── assets/                ←   下载的图片、附件
│
├── wiki/                      ← LLM 生成的百科页面（LLM 全权维护）
│   ├── overview.md            ←   整体综述 / 阶段性综合
│   ├── entities/              ←   人物、组织、项目等实体
│   ├── concepts/              ←   核心概念、术语、理论框架
│   ├── sources/               ←   每份资料的摘要
│   ├── comparisons/           ←   跨来源对比分析
│   └── queries/               ←   有价值的问答归档
│
├── index.md                   ← 内容目录（LLM 自动更新）
├── log.md                     ← 操作日志（LLM 自动记录）
├── CLAUDE.md                  ← Schema 配置文件（定义规范与工作流）
└── README.md                  ← 本文件
```

## 工作流定义

### 1️⃣ Ingest（资料摄入）

#### 加入资料

把资料放进对应的 `raw/` 子目录：

| 来源类型 | 存放位置 |
|---------|---------|
| 网页文章 / 博客 | `raw/articles/` |
| 学术论文 / 技术报告 | `raw/papers/` |
| 视频 / 播客转录 | `raw/transcripts/` |
| 已有笔记导入 | `raw/notes/` |
| 图片 / 附件 | `raw/assets/` |

> **小技巧**：使用 [Obsidian Web Clipper](https://obsidian.com/clipper) 浏览器扩展，一键将网页保存为 markdown 到 `raw/articles/`。

#### 执行 Ingest

对 LLM 说类似下面的话：

- *"帮我处理 `raw/articles/xxx.md`"*
- *"我刚放了一篇新论文在 `raw/papers/` 里，请 ingest"*

LLM 会按以下流程操作：

```
① 读取资料  →  ② 与你讨论  →  ③ 写摘要  →  ④ 更新关联页面  →  ⑤ 更新索引  →  ⑥ 记录日志
```

具体步骤：

1. **读取** `raw/` 下的原始资料文件
2. **讨论**：与你讨论关键发现和值得注意的点
3. **写入 `wiki/sources/`**：创建或更新该资料的摘要页面
4. **更新相关页面**：找到并更新 wiki 中受影响的 entity / concept 页面（可能涉及 5-15 个页面）
5. **更新 `index.md`**：在对应分类下添加/更新链接和摘要
6. **追加 `log.md`**：新增一条日志条目

> **原则**：一次只 ingest 一份资料，与你交互后再推进下一份。

### 2️⃣ Query（查询问答）

直接问你的研究课题相关的问题：

- *"这个课题目前有哪些主要流派？"*
- *"对比一下 X 和 Y 的观点差异"*
- *"我们之前有讨论过 Z 吗？"*

LLM 按以下流程操作：

```
① 查索引  →  ② 读相关页面  →  ③ 合成答案（含引用） →  ④ 可选归档
```

1. **读取 `index.md`**，确定相关页面
2. **深入阅读**相关 wiki 页面
3. **合成答案**，引用 wiki 页面作为来源（使用 Wiki 链接 `[[页面名]]`）
4. **可选归档**：如果答案有长期价值，写入 `wiki/queries/` 并更新 index.md

---

### 3️⃣ Lint（定期健康检查）

定期要求 LLM 做健康检查：

- *"检查一下 wiki 有没有矛盾"*
- *"有没有孤儿页面？"*
- *"有没有遗漏的重要概念？"*

LLM 会检查以下项目：

| 检查项 | 说明 |
|--------|------|
| 🔴 **页面矛盾** | 不同资料之间的观点冲突，需要标注 |
| ⚠️ **过时说法** | 被新资料取代的旧观点 |
| 🔗 **孤儿页面** | 没有入链（inbound links）的页面 |
| 📝 **缺失概念** | 被反复提到但缺少独立页面的概念 |
| 🕳️ **数据缺口** | 可以补充的方向，可结合 Web 搜索 |

lint 结果写入 `log.md`，并附上修复建议。

---

### 4️⃣ 日志规范

每次操作后，LLM 会以统一格式追加到 `log.md`：

```markdown
## [2026-05-27] ingest | 文章标题
- 新建: [[wiki/sources/xxx]], [[wiki/concepts/yyy]]
- 更新: [[wiki/entities/zzz]] (新增矛盾标注)
```

每行以 `## [日期] 操作类型 | 标题` 开头，可以用 Unix 命令快速检索：

```bash
grep "^## \[" log.md | tail -5    # 查看最近 5 条操作
```

## 完整 Ingest 示例

```
用户: "我刚把一篇关于 RLHF 的文章存到 raw/articles/ 里了，帮我处理一下。"

LLM:
1. 读取 raw/articles/ 下的新文章
2. 与你讨论关键发现
3. 创建 wiki/sources/rlhf-overview.md
4. 更新 wiki/concepts/reinforcement-learning.md、wiki/entities/openai.md 等
5. 在 index.md 的 Sources 和 Concepts 分类下添加条目
6. 在 log.md 追加: ## [2026-05-27] ingest | RLHF 概述
```

## 最佳实践

| 做法 | 说明 |
|------|------|
| **一次 ingest 一份资料** | 与 LLM 逐份讨论，保证质量 |
| **Obsidian Graph View** | 经常看图谱，了解知识连接情况 |
| **迭代 CLAUDE.md** | 随着使用优化工作流和页面模板 |
| **安装 Dataview 插件** | 利用 frontmatter 自动生成动态表格 |
| **版本控制** | wiki 本身就是 git 仓库，享受免费版本历史 |

## 为什么这样设计

人类维护知识库最痛苦的是"书工作"——更新交叉引用、保持摘要最新、标注矛盾、维护一致性。规模越大越难坚持。

LLM **不会厌倦、不会忘记更新引用、一次能改 15 个文件**。维基保持健康是因为维护成本几乎为零。

你的工作是：**策划资料、引导分析、提出好问题、思考意义。**
LLM 的工作是：**其余一切。**
